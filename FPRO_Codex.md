## Section Content Rules

### 01 Cover
- FANTECH/Finecoustic brand treatment
- Product category
- Product name
- Model tag
- Product tagline
- Core Selling Points list
- SKU/color block
- Product hero visual area

Cover copy should be short, confident, and product-positioning focused.

### 02 Product Features
- Keep the reference structure: hero feature first, then core feature cards.
- Hero feature should explain the product's strongest differentiator.
- Feature cards should be benefit-led, not spec-only.
- Use concise card titles and practical buyer-facing descriptions.

Reference tone examples:
- "2-in-1 Modular Design"
- "Built-in TFT + Smart Knob"
- "Programmable LED Matrix"
- "8000 mAh / 200-Hour Battery"
- "Tri-Mode Connectivity"
- "Gasket + 5-Layer Dampening"

### 03 Why Choose It
- Use target-user columns.
- Each target user gets clear pain points and product-side solutions.
- Keep the structure practical and sales-oriented.
- Do not write vague motivational copy.

Typical groups:
- Premium Gamers & Enthusiasts
- Streamers & Content Creators
- Distributors & Retailers

Adjust the group names only when the product category requires it.

### 04 Software Control
- Describe the software ecosystem only from provided facts.
- Use module blocks when the product has multiple software areas.
- For each module, include what it controls and why it matters commercially.
- If software screenshots are missing and the structure requires them, ask before rendering.

Typical module logic:
- Matrix Display Editor
- TFT Screen Editor
- Lighting / Keymap / Macro Editor
- Companion App / Driver Control

### 05 Use Scenes
- Use 6 scene cards unless the user explicitly gives a different count.
- Each scene needs: scene tag, scene title, short use-case description.
- Scenes should show where the product sells, not generic lifestyle filler.

Typical scene directions:
- Enthusiast / DIY
- Streamer / Creator
- Creative Workspace
- After Hours Gaming
- Lifestyle / Collector
- Distributor / Retail Demo

### 06 Product Comparison
- Preserve the comparison table structure.
- Do not invent competitor facts.
- If competitor data is missing, mark it as `TBD` or ask the user, depending on the user's instruction.
- Product-side facts may be filled only from user-provided specs.

### 07 Box Contents
- Preserve in-box contents, SKU, and packaging info structure.
- Do not invent packaging facts.
- Do not invent box size, carton size, weight, carton quantity, cable length, or accessories.
- If the user says `TBD`, use `TBD`.

### 08 Compatibility
- Preserve wired and wireless platform grids.
- Use only confirmed connection modes.
- Include the tri-mode advantage callout only when the product supports tri-mode.
- Do not invent platform support.
